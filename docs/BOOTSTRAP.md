# Partner Bootstrap Bundle

Partners can skip historical sync by loading a pre-built snapshot of the
indexer's consensus tables through a target block height (e.g. 956,000).
This document covers both sides: how to **export** a bundle from a fully-
synced indexer, and how a partner indexer **imports** it on first start.

## What's in (and out of) a bundle

Included (sufficient to reproduce canonical state up to `target_block`):

| Bucket | Tables |
| --- | --- |
| Block-keyed (filtered to `block_index <= target_block`) | `blocks`, `transactions`, `StampTableV4`, `SRC20`, `SRC20Valid`, `SRC101`, `SRC101Valid`, `recipients`, `reorg_events` |
| Reference (full dump) | `collections`, `collection_creators`, `collection_stamps` |

Excluded — repo-shipped CSV bootstraps populate these on first run (via `import_bootstrap_data` in `initialize_tables()`):

- `creator` (loaded from `indexer/bootstrap/creator.csv`)
- `srcbackground` (loaded from `indexer/bootstrap/srcbackground.csv`)

Excluded — derived state, rebuilt by `initialize()` after import:

- `balances` (reconstructed from `SRC20Valid` via `rebuild_balances`)
- `owners` (reconstructed from `SRC101Valid` via `rebuild_owners`)

Excluded — operator-specific or regenerated, not needed by partners:

- `src101price` — pricing data, refreshed from external sources
- `s3objects` — per-deployment image upload tracking
- `node_version_history` — per-indexer operational metadata
- `src20_metadata` — rebuilt organically as partner forward-syncs

Excluded — regenerated by the market-data scheduler after import:

- `stamp_market_data`, `src20_market_data`, `stamp_holder_cache`, `collection_market_data`, `market_data_sources`, `stamp_sales_history`

## Producing a bundle (exporter)

Run from a fully-synced indexer host with read access to the production DB:

```bash
cd indexer
RDS_HOSTNAME=… RDS_USER=… RDS_PASSWORD=… RDS_DATABASE=btc_stamps \
poetry run python tools/export_db_bootstrap.py \
    --target-block 956000 \
    --out bootstrap_956000.sql.zst
```

> Requires a MySQL-8 `mysqldump` client (it passes `--column-statistics=0`)
> and the `zstd` binary on `PATH`.

The output is a single `zstd`-compressed SQL stream — ~500 MiB through
block 956,000. Streaming is end-to-end: no multi-GB intermediate.

Then distribute. Two reasonable hosts:

- **S3** (current convention): `aws s3 cp bootstrap_956000.sql.zst s3://stampchain.io/stamps/bootstrap/ --content-type application/zstd` (publicly served as `https://stampchain.io/stamps/bootstrap/…`). Upload the `.sha256` sidecar alongside it.
- **GitHub Releases**: attach as a release asset on a tag like `bootstrap-956000`

Either way, the consuming indexer needs the bundle on a **local filesystem
path** — the import path doesn't download. Partners pre-stage the file
(via `aws s3 cp` or `wget`) then point the indexer at it.

## Importing on a partner indexer

Set two env vars in the indexer's `.env`:

```env
BOOTSTRAP_ON_EMPTY=true
BOOTSTRAP_FILE=/var/lib/btc_stamps/bootstrap_956000.sql.zst
```

On startup, the indexer:

1. Checks whether the `blocks` table is empty.
2. If yes (and both env vars set), streams `zstdcat $BOOTSTRAP_FILE | mysql` into the database.
3. After import, queries every `CHECKPOINTS_MAINNET` block ≤ target and compares `txlist_hash` / `ledger_hash` against the canonical values shipped in `index_core/check.py`.
4. On any mismatch, logs a critical error and exits with code **5**. Systemd should not auto-restart on a corrupted bootstrap — wire that into the service unit if needed (`RestartPreventExitStatus=5`).

If the DB is already populated, both env vars are no-ops — safe to leave set.

## Verification details

The check uses the hardcoded `CHECKPOINTS_MAINNET` table in
`indexer/src/index_core/check.py`, which is the same set the running
indexer would later verify against during normal operation. So
"passes bootstrap verification" is equivalent to "passes the same
consensus check the indexer would have applied while syncing those
blocks itself" — no separate trust assumption introduced.

Checkpoints past the bootstrap's `target_block` are skipped (the bundle
doesn't claim to cover them). Once the indexer starts normal forward
sync after import, those later checkpoints get checked the usual way.

## Operator notes

- **Bundle freshness**: the snapshot reflects a single point-in-time DB
  state. After production has advanced 10+ blocks past the snapshot, the
  partner will need ~10 blocks of forward sync after import — typically
  seconds.
- **Re-importing on a non-empty DB**: not supported by this path.
  Partners who want to re-import must `DROP DATABASE btc_stamps` + recreate
  the empty schema first.
- **Schema drift**: if the exporter and importer run on different code
  versions with diverged schemas, the post-import `apply_schema_updates`
  call in `initialize_tables` reconciles. If the schemas are too divergent,
  the import itself may fail — keep exporter and importer versions close.
