# Test Fixtures Guide

## Overview

To improve test coverage in CI/CD environments without requiring a Bitcoin node, we use fixture-based tests that provide real Bitcoin data for testing.

## Fixture Files

### 1. `tests/fixtures/test_data.json`
- General test data including sample transactions and blocks
- Used by parser tests
- Generated by: `poetry run python tools/debug/fetch_test_fixtures.py`

### 2. `tests/fixtures/bitcoin_node_fixtures.json`
- Special transactions for edge case testing
- Block 700000 data for Rust parser tests
- Generated by: `poetry run python tools/debug/fetch_special_test_fixtures.py`

## Fixture-Based Test Files

### Tests that NO LONGER require Bitcoin node:
1. **`test_special_txs_with_fixtures.py`** - Tests special transaction parsing
2. **`test_rust_parser_with_fixtures.py`** - Tests Rust parser functionality

### Original tests (still require Bitcoin node):
1. **`test_special_txs.py`** - Marked with `@pytest.mark.requires_bitcoin_node`
2. **`test_rust_parser.py`** - Marked with `@pytest.mark.requires_bitcoin_node`

## Running Tests

### With fixtures (no Bitcoin node needed):
```bash
# Run all tests including fixture-based ones
poetry run coverage

# Run only fixture-based tests
poetry run pytest tests/test_special_txs_with_fixtures.py tests/test_rust_parser_with_fixtures.py
```

### Excluding Bitcoin node tests:
```bash
# Run coverage excluding tests that need Bitcoin node
poetry run coverage --exclude-markers "requires_bitcoin_node"

# This is what CI uses when no Bitcoin node is available
poetry run coverage-quick
```

## Generating/Updating Fixtures

When Bitcoin transaction data changes or new test cases are needed:

```bash
# Generate general test fixtures
poetry run python tools/debug/fetch_test_fixtures.py

# Generate special transaction fixtures
poetry run python tools/debug/fetch_special_test_fixtures.py
```

**Note**: You need access to a Bitcoin node to generate fixtures.

## CI/CD Considerations

1. The CI workflow (`coverage.yml`) runs `poetry run coverage --all-formats`
2. This includes fixture-based tests but skips tests marked with `requires_bitcoin_node`
3. Fixture files are checked into the repository to ensure consistent test data
4. Total coverage should meet the 55% threshold even without Bitcoin node tests

## Best Practices

1. **Prefer fixture-based tests** over tests requiring infrastructure
2. **Keep fixtures up-to-date** with relevant Bitcoin data
3. **Document fixture purpose** in the test files
4. **Use descriptive names** for fixture data
5. **Check fixture file size** - large fixtures can slow down tests

## Migration Strategy

To migrate a test from requiring Bitcoin node to using fixtures:

1. Identify the Bitcoin data needed by the test
2. Add that data to fixture generation scripts
3. Create a new test file with `_with_fixtures` suffix
4. Load fixtures in `setUpClass` method
5. Replace RPC calls with fixture data access
6. Verify test still validates the same functionality
7. Keep original test for environments with Bitcoin node
