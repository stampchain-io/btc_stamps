# Multi-Source Market Data Cache System - Product Requirements Document

## Executive Summary

The Bitcoin Stamps indexer frontend explorer currently suffers from severe performance issues due to excessive external API calls to counterparty.io and other exchange APIs. Collection pages take 10+ seconds to load, make 40+ concurrent API calls, and frequently encounter "dispatch task is gone" errors. This PRD outlines the implementation of a comprehensive market data cache system that will reduce page load times to under 2 seconds while supporting both Bitcoin Stamps (art) and SRC-20 tokens from multiple data sources.

## Problem Statement

### Current Performance Issues
- Collection pages load in 10+ seconds due to excessive API calls
- 40+ concurrent calls to counterparty.io for floor prices and holder data
- "dispatch task is gone" errors from external API overload
- Holder filtering requires fetching ALL holder data for EVERY stamp via paginated API calls
- SRC-20 tokens require different data sources (OpenStamp, StampScan, KuCoin) not currently integrated
- 95% of external API calls are redundant and could be cached

### Business Impact
- Poor user experience leading to user abandonment
- Unreliable frontend performance affecting platform credibility
- Inability to implement advanced filtering features due to performance constraints
- Limited scalability for growing stamp and SRC-20 token ecosystems

## Solution Overview

Implement a unified multi-source market data cache system that:
- Pre-computes floor prices, holder counts, and volume metrics for both stamps and SRC-20 tokens
- Aggregates data from multiple sources with confidence-weighted scoring
- Updates cache via background jobs every 15-30 minutes (stamps) and 2-10 minutes (SRC-20)
- Provides instant database queries instead of external API calls
- Supports real-time updates for high-activity assets

## Technical Requirements

### Database Schema Enhancement
- **stamp_market_data**: Floor prices, holder counts, volume data for stamps
- **src20_market_data**: Multi-exchange market data for SRC-20 tokens  
- **market_data_sources**: Source attribution and reliability tracking
- **collection_market_data**: Aggregated collection-level metrics
- **stamp_holder_cache**: Detailed holder information for filtering

### Multi-Source Data Integration
- **Stamps**: Counterparty API (dispensers, balances, sends)
- **SRC-20 Tokens**: OpenStamp API, StampScan API, KuCoin API, future exchanges
- **Aggregation Logic**: Confidence-weighted averaging with conflict resolution
- **Source Reliability**: Health monitoring and quality scoring (0-10 scale)

### Background Job System
- **Adaptive Frequencies**: Stamps (15-30 min), SRC-20 (2-10 min based on volatility)
- **Batch Processing**: 50 stamps or 20 SRC-20 tokens per batch with concurrency control
- **Rate Limiting**: Respect external API limits with exponential backoff
- **Real-time Events**: Immediate updates for dispenser events and external trades

### API Enhancement
- **Enhanced Controllers**: Serve cached data instead of live API calls
- **Filter Support**: Enable holder count and floor price filtering from database
- **Collection Aggregation**: Pre-computed collection statistics
- **Health Endpoints**: Monitor cache freshness and source reliability

### Frontend Integration
- **Zero API Calls**: Collection pages load from cached data only
- **Advanced Filtering**: Holder count ranges, floor price ranges, volume filters
- **Real-time Updates**: WebSocket support for high-activity assets
- **Fallback Handling**: Graceful degradation when cache is stale

## Risk Mitigation

### Data Freshness Risk
- **Mitigation**: Adaptive update frequencies based on asset activity
- **Monitoring**: Cache age alerts and freshness dashboards
- **Fallback**: Real-time API calls for critical high-activity assets

### Multi-Source Conflicts
- **Mitigation**: Confidence-weighted aggregation with source reliability scoring
- **Validation**: Cross-source data consistency checks
- **Resolution**: Automated conflict detection and manual review processes

### External API Failures
- **Mitigation**: Multi-source redundancy and graceful degradation
- **Monitoring**: Source health tracking and automatic failover
- **Recovery**: Retry logic with exponential backoff and circuit breakers

### SRC-20 Volatility
- **Mitigation**: Adaptive scheduling based on price changes and volume
- **Processing**: Priority queues for high-activity tokens
- **Efficiency**: Batch updates with intelligent grouping
